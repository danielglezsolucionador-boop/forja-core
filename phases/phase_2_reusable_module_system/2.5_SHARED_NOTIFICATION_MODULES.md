# 2.5 SHARED NOTIFICATION MODULES

## PURPOSE
Define reusable notification modules for operational messages and app events.

## OBJECTIVES
- Standardize notification payloads.
- Separate routing, formatting, and delivery concerns.
- Prepare provider-specific channels without locking architecture.

## REQUIRED MODULES
- Notification payload schema.
- Severity and category mapping.
- Channel routing boundary.
- Delivery result contract.
- Retry and failure audit concept.

## RULES
- Do not send real notifications from shared module definition.
- Do not hide delivery failures.
- Do not hardcode one ecosystem app as the only consumer.

## ECOSYSTEM COMPATIBILITY
- Notification modules must support Telegram, email, webhooks, and dashboard alerts later.
- Message contracts must be reusable by FORJA-generated systems.

## SUCCESS CONDITIONS
- Notification modules have clear contracts.
- Provider delivery remains pluggable.
- Failure handling is visible by design.
