# 2.3 SHARED AUTH MODULES

## PURPOSE
Define reusable authentication modules for internal tools and future generated apps.

## OBJECTIVES
- Standardize JWT validation behavior.
- Separate auth primitives from app-specific authorization.
- Keep auth secure, minimal, and auditable.

## REQUIRED MODULES
- Token creation boundary.
- Token validation boundary.
- Password hashing helper.
- Current-user dependency concept.
- Protected route pattern.

## RULES
- Do not add advanced roles before governance approves them.
- Do not store secrets in generated code or documentation.
- Do not bypass validation for development convenience.

## ECOSYSTEM COMPATIBILITY
- Auth modules must be usable by dashboards, APIs, and admin surfaces.
- Auth contracts must remain provider-neutral.

## SUCCESS CONDITIONS
- Shared auth responsibilities are defined.
- Advanced authorization remains separate.
- Reuse does not weaken security.
