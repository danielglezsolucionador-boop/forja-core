# 2.1 SHARED BACKEND MODULES

## PURPOSE
Define reusable backend module standards for FORJA-generated and ecosystem services.

## OBJECTIVES
- Identify backend primitives that should be shared.
- Keep modules independent, small, and testable.
- Prevent copy-paste backend foundations across apps.

## REQUIRED MODULES
- Config/settings module.
- Logging module.
- Health module.
- Error handling module.
- Repository and service boundary patterns.

## RULES
- Do not create framework abstractions without repeated need.
- Do not couple shared modules to one application.
- Each module must have a clear contract and validation path.

## ECOSYSTEM COMPATIBILITY
- Modules must be usable by FORJA, Hermes-adjacent tools, and future client apps.
- Shared backend modules must not assume a specific deployment provider.

## SUCCESS CONDITIONS
- Shared backend scope is explicit.
- Modules can be adopted incrementally.
- No hidden runtime architecture is introduced.
