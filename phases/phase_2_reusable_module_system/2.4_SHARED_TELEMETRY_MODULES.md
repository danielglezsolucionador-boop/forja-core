# 2.4 SHARED TELEMETRY MODULES

## PURPOSE
Define reusable telemetry modules for operational evidence across FORJA outputs.

## OBJECTIVES
- Standardize basic telemetry capture.
- Support debugging without leaking sensitive data.
- Prepare dashboards to consume real operational signals.

## REQUIRED MODULES
- Event emitter boundary.
- Error telemetry boundary.
- Request timing boundary.
- Correlation identifier concept.
- Redaction rules.

## RULES
- Do not log tokens, passwords, private prompts, or credentials.
- Do not create telemetry that cannot be queried or validated.
- Keep instrumentation minimal until operational value is proven.

## ECOSYSTEM COMPATIBILITY
- Telemetry modules must support shared dashboards and audit flows.
- Signal names should remain consistent across generated apps.

## SUCCESS CONDITIONS
- Telemetry has a reusable module shape.
- Sensitive data handling is explicit.
- Future dashboards can rely on consistent signals.
