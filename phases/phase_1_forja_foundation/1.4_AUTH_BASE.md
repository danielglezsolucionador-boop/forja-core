# 1.4 AUTH BASE

## PURPOSE
Define the authentication foundation for FORJA with JWT as the initial reusable auth pattern.

## OBJECTIVES
- Establish a minimal real auth contract.
- Separate auth foundation from advanced identity features.
- Keep future auth modules reusable across ecosystem apps.

## REQUIRED COMPONENTS
- JWT token contract.
- Password hashing requirement.
- Auth dependency or middleware boundary.
- Protected endpoint validation concept.
- Basic user model boundary.

## RULES
- Do not add multi-tenant auth, refresh token rotation, roles, or permissions before authorization.
- Do not fake users or bypass token validation.
- Auth must be explicit, testable, and reversible.

## ECOSYSTEM COMPATIBILITY
- Auth base must support future dashboards, APIs, and internal tools.
- Token behavior must be suitable for shared backend modules.

## SUCCESS CONDITIONS
- Auth scope is limited and real.
- Required endpoints and validation expectations are known.
- Advanced identity remains out of scope until later phases.
